# Frontend Rules ŌĆö AI Reel Creator & Editor

Applies to everything under `components/`, `app/` (client-side), and `lib/hooks/`.

---

## Component Model

### Server vs Client
- **Default to Server Components.** Only add `"use client"` when you need:
  - React hooks (`useState`, `useEffect`, `useRef`, `useReducer`, etc.)
  - Browser APIs (`window`, `localStorage`, `navigator`, etc.)
  - Event handlers attached to interactive elements
  - Vercel AI SDK's `useChat` or any `@ai-sdk/react` hook
- Place `"use client"` at the very top of the file, before all imports.
- Never put data-fetching logic inside a `"use client"` component ŌĆö fetch in a Server Component parent and pass as props.

### Component location rules
```
components/ui/          ŌåÉ shadcn-generated primitives ONLY. Never add custom logic here.
components/chat/        ŌåÉ Chat UI, message bubbles, streaming indicators
components/editor/      ŌåÉ Reel timeline, script editor, caption editor, preview panel
components/layout/      ŌåÉ AppShell, Sidebar, TopNav, PageHeader
app/(dashboard)/        ŌåÉ Route-level page components (thin wrappers)
```

- One component per file. File name = component name in kebab-case (`reel-preview.tsx`).
- Export the component as a named export AND as the default export for page files.

---

## shadcn/ui

- **All new UI primitives** must come from shadcn/ui. Run `npx shadcn add <component>` ŌĆö never hand-roll equivalents.
- Never edit files in `components/ui/` ŌĆö they are regenerated by shadcn. Put customisation in wrapper components.
- Use the `variant` and `size` props provided by shadcn instead of overriding with inline styles.
- For compound UI patterns (e.g., a dialog with a form), create a wrapper in `components/<domain>/` that composes shadcn primitives.

---

## Styling

- Use Tailwind utility classes exclusively. No inline `style={{}}` except for dynamic values that cannot be expressed as utilities (e.g., CSS custom property injection for waveform widths).
- `cn()` from `lib/utils.ts` for every conditional or merged class string ŌĆö never raw template literals.
- Design tokens (`--primary`, `--muted`, `--background`, etc.) are defined in `app/globals.css`. Use them as `bg-primary`, `text-muted-foreground`, etc.
- Dark mode is `.dark` class on `<html>` ŌĆö never use `dark:` media query variants.
- Animations: use `tw-animate-css` utilities (`animate-pulse`, `animate-fade-in`, etc.) ŌĆö do not add custom keyframes unless unavoidable.
- For responsive layouts, follow mobile-first: base styles for mobile, `md:` and `lg:` for wider breakpoints.

---

## State Management

### Local state (preferred for isolated UI state)
- `useState` for toggle, input value, loading flags.
- `useReducer` when a component manages more than 3 related state fields.

### Shared / persistent state
- Sessions and messages: already handled by `useChat` + Supabase.
- For global UI state (sidebar open, selected reel, active tool): use React Context in a `lib/context/<feature>-context.tsx` file.
- Do NOT add Zustand, Jotai, or any external state library without explicit approval ŌĆö React Context is sufficient at current scale.

### Chat / AI state
- `useChat` from `@ai-sdk/react` is the single source of truth for messages and streaming state.
- Always pass transport via `useMemo` so it only rebuilds when `sessionId` changes (see existing `chat.tsx` pattern).
- Use `status === 'streaming'` to gate input and show loading UI ŌĆö never use a separate `isLoading` boolean.
- `setMessages([])` resets the chat; call `createSession()` immediately after to start a fresh Supabase session.

---

## Forms & Input

- Use controlled inputs bound to `useChat`'s `handleInputChange` / `input` pair for chat forms.
- For non-chat forms (reel metadata, upload, settings), use React `useState` + validation on submit ŌĆö do not add `react-hook-form` without explicit approval.
- Disable all interactive elements while `status === 'streaming'` or while async operations are pending.
- Always trim and validate user input before sending to the API.

---

## Custom Hooks

- Place shared hooks in `lib/hooks/use-<feature>.ts`.
- Each hook must have a single responsibility. Split complex data-fetching from UI state.
- Hooks that call Supabase from the browser must use `createClient` from `lib/supabase/client.ts`.
- Always handle loading and error states in hooks, not in the component.

```ts
// Example shape for data-fetching hooks
function useSessionList() {
  const [sessions, setSessions] = useState<Session[]>([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<string | null>(null)
  // ...
  return { sessions, loading, error }
}
```

---

## Performance

- Wrap expensive child renders in `React.memo` only when profiling shows a real problem ŌĆö do not pre-optimise.
- `useMemo` and `useCallback` only for values/callbacks that are passed as props to memoised children or used as `useEffect` dependencies.
- Never call `useEffect` without a dependency array ŌĆö always specify deps explicitly.
- Images: use `next/image` with explicit `width`/`height` or `fill` + a sized container. Never raw `<img>`.
- For lists of messages or reels, always provide a stable `key` (use the DB `id` field, not array index).

---

## Accessibility

- Interactive elements must be keyboard-accessible ŌĆö use semantic HTML (`<button>`, `<input>`, `<form>`).
- Every `<img>` and `next/image` must have a descriptive `alt` attribute.
- Use `aria-label` on icon-only buttons.
- Focus management: when opening a dialog/modal, focus the first interactive element inside it (shadcn Dialog handles this automatically).

---

## Icons

- Use `lucide-react` exclusively. Import individual icons ŌĆö never the whole library.
- Size icons with Tailwind (`w-4 h-4`, `w-5 h-5`) ŌĆö do not use the `size` prop with pixel values.

---

## TypeScript in Components

- No `any` types. Use the `Database` type from `lib/supabase/types.ts` for all Supabase row shapes.
- Derive prop types from data types: `type SessionCardProps = { session: Database['public']['Tables']['sessions']['Row'] }`.
- Use `React.FC` only if the component accepts children with a defined type. Otherwise, use plain function signatures.
- All event handler parameters must be typed (`React.FormEvent<HTMLFormElement>`, `React.ChangeEvent<HTMLInputElement>`).
